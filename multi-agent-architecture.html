Mini-OpenClaw 多Agent架构

LLM驱动的智能任务分发与协同执行系统

📋 概述

Mini-OpenClaw 采用LLM驱动的多Agent协同架构，实现了从用户意图理解到任务拆解、分发、执行的完整流程。系统核心创新在于使用LLM替代硬编码的关键词匹配，实现更智能的任务分析与分发决策。

核心设计理念

系统不再依赖预定义的关键词规则来判断任务类型，而是让LLM理解用户消息的语义，动态生成执行计划（ExecutionPlan），包含策略选择、任务拆解、Agent分配等决策。

🔄 执行流程详解

Step 1: 消息入口与模式检测

用户消息通过 /api/chat 接口进入系统，首先检查多Agent模式是否开启。

# backend/api/chat.py
multi_agent_mode = get_multi_agent_mode()

if multi_agent_mode:
    planner = get_task_planner()
    plan = await planner.plan_execution(message)
    
    if plan.strategy == "multi":
        # 进入多Agent执行路径
        async for event in _multi_agent_generator(...):
            yield event

Step 2: LLM任务规划

LLMTaskPlanner收集所有可用Agent的能力描述，构建Planning Prompt，调用LLM生成执行计划。

# backend/graph/llm_task_planner.py

async def plan_execution(self, message: str) -> ExecutionPlan:
    # 1. 收集Agent能力
    capabilities = collect_agent_capabilities(coordinator, base_dir)
    
    # 2. 构建Prompt
    system_prompt = _build_planner_prompt(capabilities)
    messages = [SystemMessage(system_prompt), HumanMessage(message)]
    
    # 3. LLM调用（带超时）
    response = await asyncio.wait_for(
        self.llm.ainvoke(messages), 
        timeout=10
    )
    
    # 4. 解析JSON响应
    plan = _parse_llm_response(response.content, available_agents)
    
    return plan or _fallback_plan(message)

Step 3: 执行计划结构

LLM返回的ExecutionPlan包含策略、任务列表、置信度等关键信息。

Step 4: 任务创建与Todo分解

TaskExecutor根据ExecutionPlan创建任务，并为每个Todo分配ID和执行Agent。

# backend/graph/task_executor.py

def create_task_from_plan(self, message, plan, session_id):
    task_id = f"TASK_{int(time.time())}_{uuid.uuid4().hex[:8]}"
    
    # 从plan.todos构建Todo列表
    todos = []
    for i, planned_todo in enumerate(plan.todos):
        todo = TodoItem(
            id=f"{task_id}_todo_{i}",
            content=planned_todo.content,
            agent=planned_todo.agent_name
        )
        todos.append(todo)
    
    # 动态初始化Agent状态
    involved_agents = set(t.agent_name for t in plan.todos)
    agent_status = {agent: "idle" for agent in involved_agents}
    
    return task_id

Step 5: Todo串行执行

遍历Todos列表，根据agent_name决定执行路径：Domain Agent通过TaskDispatcher分发，Primary Agent直接执行。

# backend/api/chat.py - _multi_agent_generator

for todo in todos:
    # 更新Todo状态
    task_executor.update_todo_status(task_id, todo_id, "in_progress")
    
    if todo_agent != "primary_agent":
        # Domain Agent执行 - 通过TaskDispatcher
        async for event in task_dispatcher.dispatch_task(
            task_content=task_prompt,
            target_agent=todo_agent,
            task_id=task_id
        ):
            yield event
    else:
        # Primary Agent执行
        async for event in agent_manager.astream(prompt, session_id):
            yield event
    
    # Todo完成
    task_executor.update_todo_status(task_id, todo_id, "completed")

Step 6: TaskDispatcher分发执行

TaskDispatcher获取目标Agent实例，构建专属System Prompt，执行流式推理并返回事件。

# backend/graph/task_dispatcher.py

async def dispatch_task(self, task_content, target_agent, task_id):
    # 1. 获取Agent实例
    agent = self._get_agent_instance(target_agent)
    
    # 2. 构建专属System Prompt
    system_prompt = agent.build_system_prompt()
    
    # 3. 创建LangChain Agent
    lc_agent = create_tool_calling_agent(self.llm, agent.tools, system_prompt)
    executor = AgentExecutor(lc_agent, tools=agent.tools)
    
    # 4. 流式执行
    async for event in executor.astream_events(input, version="v2"):
        if event["event"] == "on_chat_model_stream":
            yield {"type": "token", "content": token}
        elif event["event"] == "on_tool_start":
            yield {"type": "tool_start", ...}
        elif event["event"] == "on_tool_end":
            yield {"type": "tool_end", ...}

📡 SSE事件流

系统通过Server-Sent Events (SSE) 实现实时事件推送，前端可以根据事件类型动态更新UI。

事件类型 触发时机 关键数据
strategy_decided 策略决策完成 strategy, target_agent, confidence, reason
task_created 任务创建成功 task_id, message, todos, agent_status
todo_update Todo状态变更 todo_id, old_status, new_status, agent, result
agent_status Agent状态变更 agent_name, old_status, new_status
token LLM输出Token content, agent_name, task_id
tool_start 工具调用开始 tool, input, agent_name, task_id
tool_end 工具调用结束 tool, output, elapsed_time, tool_status
stats_update 统计数据更新 input_tokens, output_tokens, tool_calls_count
task_complete 任务执行完成 task_id, summary, final_stats
done 流式输出完成 content, session_id, tool_calls

📝 System Prompt构建

每个Agent的System Prompt由多个组件动态拼接而成，确保Agent具备正确的角色认知和行为准则。

System Prompt拼接顺序

  1. SKILLS_SNAPSHOT_LOCAL.md - Agent专属技能列表
  2. GLOBAL_MEMORY/AGENTS_GLOBAL.md - 全局行为准则
  3. {agent_name}/SOUL.md - 核心设定（人格、语气、边界）
  4. {agent_name}/IDENTITY.md - 自我认知（名称、风格）
  5. GLOBAL_MEMORY/USER.md - 用户画像
  6. {agent_name}/AGENTS_LOCAL.md - 专属行为准则
  7. {agent_name}/MEMORY.md - 专属长期记忆
  8. COORDINATION_SNAPSHOT.md - 协同状态快照（仅Primary/Coordinator加载）
# backend/graph/base_agent.py

def build_system_prompt(self) -> str:
    parts = []
    
    # 1. Agent专属技能
    parts.append(read_file("SKILLS_SNAPSHOT_LOCAL.md"))
    
    # 2. 全局行为准则
    parts.append(read_file("GLOBAL_MEMORY/AGENTS_GLOBAL.md"))
    
    # 3-4. 核心设定与自我认知
    parts.append(read_file("SOUL.md"))
    parts.append(read_file("IDENTITY.md"))
    
    # 5. 用户画像
    parts.append(read_file("GLOBAL_MEMORY/USER.md"))
    
    # 6-7. 专属准则与记忆
    parts.append(read_file("AGENTS_LOCAL.md"))
    parts.append(read_file("MEMORY.md"))
    
    # 8. 协同状态（仅Primary/Coordinator）
    if self.agent_type in [AgentType.PRIMARY, AgentType.COORDINATOR]:
        parts.append(read_file("COORDINATION_SNAPSHOT.md"))
    
    return "\n\n".join(parts)

🔄 Fallback机制

当LLM调用失败、超时或返回无效响应时，系统会自动降级到基于关键词的Fallback策略。

Fallback执行路径

def _fallback_plan(message: str) -> ExecutionPlan:
    # 使用原有的StrategySelector（关键词匹配）
    selector = get_strategy_selector()
    analysis = selector.analyze(message)
    
    if analysis.strategy == ExecutionStrategy.MULTI_AGENT:
        # 使用预定义的任务模板
        task_type = executor._analyze_task_type(message)
        template = executor.TASK_TEMPLATES.get(task_type, "general")
        
        todos = [PlannedTodo(content=item["content"], 
                             agent_name=item["agent"]) 
                 for item in template]
        
        return ExecutionPlan(strategy="multi", todos=todos, 
                           source="fallback")
    
    # 最终降级：单Agent执行
    return ExecutionPlan(strategy="single", todos=[], 
                       source="fallback")

TaskExecutor内置预定义的任务模板，用于Fallback场景下的任务分解。

  2. 执行数据统计分析 → data_agent
  3. 生成分析结果报告 → primary_agent

data_visualization 模板

  1. 分析数据结构和内容 → data_agent
  2. 执行数据统计分析 → data_agent
  3. 生成数据可视化图表 → data_agent
  4. 输出分析报告 → primary_agent

document_parsing 模板

  1. 解析文档内容 → doc_agent
  2. 提取关键信息 → doc_agent
  3. 生成处理结果 → primary_agent

general 模板

  1. 理解任务需求 → primary_agent
  2. 执行任务处理 → primary_agent
  3. 生成响应结果 → primary_agent

🔐 资源锁机制

CoordinationManager实现了基于内存的资源锁，防止多个Agent同时修改同一资源。

def acquire_lock(self, resource: str, agent: str, timeout: int = 30) -> bool:
    # 检查是否被持有
    if resource in self._locks:
        lock_time, lock_agent = self._locks[resource]
        
        # 超时检测
        if time.time() - lock_time > timeout:
            del self._locks[resource]  # 强制释放
        elif lock_agent == agent:
            return True  # 同一Agent重入
        else:
            return False  # 被其他Agent持有
    
    # 获取锁
    self._locks[resource] = (time.time(), agent)
    return True
